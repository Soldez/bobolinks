    1.6 Your Website, part 1 (medium)

    Submit Assignment
    • Due Monday by 11am
    • Points 0
    • Submitting a website url

    Summary

    Up to this point, you have been working with existing repositories. In this challenge, you'll be creating your own repository that will host your very own website! You'll be using a GitHub repository and GitHub Pages (Links to an external site.) to set it up.

    Release 0: Create a website repository

    1.Create a new repository (Links to an external site.)

    This will take you to the new repository form you've already seen.

    1. Fill out the new repository form.

      • You must, must, must name your repository "[USERNAME].github.io".Your repository should be public.
      • Check "Initialize this project with a README" and add an MIT license.

      I'm gonna say it again: You must, must, must name your repository [USERNAME].github.io. If you have trouble, double check you spelled it correctly.

    2. Click on the "Create Repository" button.

     

    Do a celebration dance!

    Release 1: Clone the repository

    Now you'll want to clone the repository to your computer like you did before. Go through the cloning workflow (not the forking part) you created in the 1.4 Create, Fork, and Clone Repositories challenge. Before you clone, make sure you are not in your phase-0 repository in your command line. 

    The first step is to get the URL of the repo. This is found here:

    Your files should look like this if you set them up correctly.

    (NOTE: DBC-Stuff is a random folder I created. these can be on your desktop or in another folder, just make sure they are not inside one another.)

    Release 2: Open Source and Licenses

    You added an MIT License to your repository. Read these articles to explain why:

    • Open Source (Links to an external site.)
    • Open Source licensing (Links to an external site.)
    • A short guide to Open Source (Links to an external site.)

    Release 3: Your temporary index

    Now you will create a temporary index for your personal website. You will be using Sublime Text (Links to an external site.) as your text editor (Links to an external site.). Review the Perfect Workflow in Sublime Text (Links to an external site.) if you want some help on how to write code using Sublime.

    • Because you are starting a new challenge, you should checkout a new branch to start working on before doing anything else. 
    • Open a new file in Sublime
    • Save the file as index.html in the [USERNAME].github.io directory.

    • Add the following HTML code to your file:
    <html>
  <body>
      Hello DBC!
  </body>
</html>
    • Save the file

    Release 4: Commit and Push

    Using your git skills:

    • Add the file to the stage.
    • Commit your changes.
    • Push your feature branch to GitHub.
    • Create a Pull Request to merge your feature branch to the master branch.
    • Once it's merged, checkout your master branch.
    • Pull your new master branch down to your local machine.
    • Checkout your master branch.
    • Delete the feature branch on your local computer. (If it gives you a warning heed it!)

    After your push is done, your remote repo which lives on github.com will be updated. Go to http://[USERNAME].github.io to see your HTML page live!

    NOTE: It can sometimes take up to 15 minutes to update the site after creating your repo, so if you run into a 404 error, be patient and check again later. If it's still not up, double check you spelled your username correctly in the repository name.

    You can also see the new index.html file in your GitHub repository!

    Release 5: Reflect

    Now's the time to reflect on your learning! Remember to create a feature branch! Then, download this reflection file Link and move it into the week-1 directory in your phase-0 repository using only the command line! Then, open the file in sublime and add your reflection to the file. Using your new git and GitHub skills, commit and push your changes to your remote repository and go through the pull request process! Make sure you can see your reflection on GitHub. 

     

    Release 6: Peer Review (to complete next week)

    When peer reviewing, please look at the longer descriptions for help in determining what score a person should receive as well as the Peer Reviewing Instructions and Guidance.
